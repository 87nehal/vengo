# vengo

`vengo` is an early Go-native application framework inspired by Spring Boot's developer experience, with a smaller and more explicit runtime model.

The beta implementation includes:

- core application lifecycle and module registration
- named service registry with typed lookup helpers
- TOML/JSON configuration with typed struct binding, profile support, and redacted reports
- small auto-configuration registry
- `net/http` based web module with route groups, middleware, structured errors, and JSON helpers
- actuator-style health, info, metrics, logging, and telemetry modules
- data access support for `database/sql`, transactions, migrations, health checks, and slow query logging
- composable security middleware for API keys, JWT, sessions, secure headers, CORS, and CSRF
- testing helpers for in-process apps, dependency overrides, HTTP requests, and temporary config
- CLI scaffold for version, project generation, config inspection, and dependency graph display
- runnable examples for web, data, security, CLI workers, and testing

The project is intentionally one Go module for now. Splitting packages into separate modules can happen once the public API is more stable.

## Quick Start

```bash
go test ./...
go run ./examples/hello
```

Then open:

```text
http://localhost:8080/
http://localhost:8080/actuator/health
```

The hello example returns a small text response at `/` and actuator health returns `{"status":"UP"}` when all checks pass.

## CLI

```bash
go run ./cmd/vengo version
go run ./cmd/vengo new orders-api github.com/you/orders-api --modules=web,data
go run ./cmd/vengo config
go run ./cmd/vengo deps
go run ./cmd/vengo routes
go run ./cmd/vengo doctor
go run ./cmd/vengo run .
```

If the module path is omitted, `vengo new` uses the project directory name as the module path.
`vengo config` prints the resolved local configuration with secrets redacted. `vengo deps` prints a dependency graph from a `vengo-deps.json` file generated by an app. `vengo routes` prints registered HTTP routes from a `vengo-routes.json` file. `vengo doctor` checks the Go toolchain, module path, framework version, and active profile. `vengo run .` watches the target directory and rebuilds/restarts on `.go`, `.toml`, `.json`, `.yaml`, and `.yml` changes.

The `--modules` flag for `vengo new` accepts a comma-separated list of `web`, `data`, and `auth`. The `web` module is selected by default. The generated `main.go` uses the typed config pattern (`config.LoadDefaults` + `config.Bind`) and produces an `application.toml` for project-level configuration.

## Health Endpoint

The actuator module exposes `/actuator/health` by default:

```go
app := core.New("hello", web.New(":8080"), actuator.NewHealth())
```

Use options when you need a custom path or explicit checks:

```go
health := actuator.NewHealth(
	actuator.WithPath("/healthz"),
	actuator.WithChecks(actuator.Check{Name: "self", Check: func(context.Context) error { return nil }}),
)
```

## Configuration

Load `application.toml` (or `.json`) with profile overrides and bind into typed structs:

```go
type ServerConfig struct {
    Port int    `config:"server.port" default:"8080"`
    Host string `config:"server.host" default:"localhost"`
}

cfg, _ := config.LoadDefaults(ctx, config.ActiveProfile())

var serverCfg ServerConfig
config.Bind(cfg, &serverCfg)
```

Source precedence (later overrides earlier):

1. `application.toml` (from `.` or `./config/`)
2. `application-{profile}.toml`
3. Environment variables with `APP_` prefix (`APP_SERVER_PORT` → `server.port`)

Set the active profile via `APP_PROFILE` or `VENGO_PROFILE` environment variables.

## Design Direction

- Prefer explicit registration over package scanning and hidden auto-wiring.
- Keep the core dependency-light.
- Build diagnostics into every major subsystem.
- Make integrations optional modules rather than framework requirements.
